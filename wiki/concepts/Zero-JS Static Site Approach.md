# Zero-JS Static Site Approach

## Definition

The zero-JS static site approach is the shared design philosophy of [[GhostKellz]]'s
three web properties: render everything to plain HTML and CSS at build time, ship no
client-side framework or hydration runtime, and add JavaScript only where a specific
interaction genuinely requires it. The result is sites that load instantly and deploy
as flat files.

## How It Plays Out Across the Sites

| Site | JS posture |
|------|------------|
| [[ckelley.dev]] | **No client JS framework at all.** Diagrams animate via CSS; scroll-spy and reveal use small native scripts ( `IntersectionObserver`). |
| [[ghostkellz.sh]] | Static HTML + a thin layer of [[Alpine.js]] for copy buttons, key reveal, and a terminal replay. |
| [[cktechx.com]] | "Zero JS by default" Astro output; [[Alpine.js]] only for the mobile nav and small dynamic features. |

## Why This Approach

> [!key-insight]
> Static-first removes the runtime: no server process, no hydration, no charting or
> SPA library to ship or patch. Pages are flat files behind [[NGINX Static Site
> Hosting]], so deploys are an `rsync` and there is essentially no attack surface
> beyond the web server itself.

A telling demonstration is [[ckelley.dev]]'s architecture page: even complex,
branching topology graphs are rendered as [[Hand-Authored Inline SVG Diagrams]] with
CSS-only animation — proving that "no JS" need not mean "no rich visuals."

## Trade-offs

- Interactivity must be added manually and kept minimal (hence [[Alpine.js]] rather
  than React/Vue).
- Rich diagrams are hand-authored rather than generated by a library — more upfront
  effort, but full control and zero runtime dependency.
